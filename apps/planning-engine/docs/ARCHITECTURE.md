## 1. Overview

Ordo v2 is a manufacturing production scheduling system for an industrial site producing BDH (Boues De forage / Huiles) products. It handles:

- **Order management**: MTS (Make To Stock), MTO (Make To Order - ALDES), NOR (Normal)
- **Component feasibility**: Recursive BOM checking down to purchased components
- **Production scheduling**: Day-by-day, line-by-line planning with capacity constraints
- **Decision support**: Allocation virtuelle avec gestion de la concurrence
- **Reporting**: Action reports, service-rate KPIs, kanban risk analysis, charge heatmaps

## 5. Orders And Planning (`production_planning/orders/`, `production_planning/planning/`)

### 5.1 Charge Calculator (`charge_calculator.py`)

Core recursive algorithm for computing production hours per workstation.

**`calculate_article_charge(article, quantity, loader, visited=None) → dict[str, float]`**

1. Read the article's gamme (routing) → `hours = quantity / cadence` per poste
2. For each fabricated component in nomenclature → recurse with `qty = qte_lien * quantity`
3. Purchased components terminate recursion
4. Merge results by summing hours per poste

**`calculate_weekly_charge_heatmap(besoins, loader, num_weeks) → list[ChargeByPoste]`**

Main entry point for charge visualization:
- Separates besoins into BACKLOG, EN_COURS, S+1..S+N
- Consumes forecasts before computing charge
- Returns structured heatmap data

### 5.2 Forecast Consumption (`forecast_consumption.py`)

**`consume_forecasts_by_article(besoins) → (adjusted_besoins, stats)`**

1. Separate COMMANDE and PREVISION
2. Aggregate quantities per article
3. For each article: `net_forecast = max(0, total_forecast - total_command)`
4. Return commands unchanged + synthetic besoins for net forecasts only

### 5.3 Matching (`matching.py`)

**`CommandeOFMatcher`** — Matches customer orders to production orders.

**MTS path** (`_match_mts`): Direct lookup via `OF_CONTREMARQUE` field.

**NOR/MTO path** (`_match_nor_mto`):
1. Allocate available stock first
2. If article is ACHAT → no OF needed
3. Search for best OF: Ferme > Planifie > Suggéré
4. Sort by: status priority, date proximity, quantity available
5. Support OF sharing between multiple orders via `OFConso`

**Priority sorting**: `(command_type, expedition_date, order_age)`

### 5.4 Allocation Manager (`allocation.py`)

**`AllocationManager`** — Orchestrates virtual stock allocation with concurrency management.

**`StockState`** — Virtual stock tracker (`initial_stock`, `allocated_stock`). Methods: `get_available()`, `allocate()`, `add_supply()`.

**`AllocationStatus`** (enum): FEASIBLE, NOT_FEASIBLE, SKIPPED, DEFERRED

**Algorithm**:
1. Separate FERMES with existing ERP allocations from those needing virtual allocation
2. Sort OFs by priority (date + feasibility)
3. Process each OF: create temporary `RecursiveChecker` with shared `StockState`, check feasibility, commit virtual allocations if feasible

### RecursiveChecker (`recursive.py`)

The most complex checker — **552 lines** of recursive BOM traversal.

**Core algorithm (`_check_article_recursive`)**:
1. Fetch nomenclature. If missing → alert (not blocking). If empty → feasible.
2. If parent OF is FERME → load existing ERP allocations
3. For each component:
   - **Purchased**: Check stock (virtual or real). Handle phantom articles (AFANT) → resolve to real variants.
   - **Fabricated**: Check stock first. If insufficient → search for a producing OF (ferme > planifie > suggéré) and recurse into it.
4. If no OF found for fabricated component → mark missing but still recurse BOM to identify root purchase shortages.

**Special features**:
- Phantom article resolution (`_check_phantom_component`): tries each variant until one covers 100%
- Subcontracted article handling (category starting "ST" → treat as purchase)
- Phantom sibling variant exclusion (prevents double-counting)

## 7. Scheduling (`production_planning/scheduling/`)

### 7.1 Architecture

The scheduler has **two pipelines**:

| Pipeline | Entry Point | Status |
|----------|------------|--------|
| **Active** | `engine.run_schedule()` (exposed via `__init__.py`) | Functional |
| **Legacy** | `run.run_schedule()` | Broken (missing model classes) |

Only the active pipeline is described below.

### 7.2 Active Pipeline Flow

```
engine.run_schedule(loader, reference_date, ...)
│
├─ 1. Load weights from config/weights.json
├─ 2. Build working day horizon (5 days by default)
├─ 3. Identify target lines (PP_830, PP_153) and their articles
│
├─ 4. Select candidates (_select_candidates_from_matching)
│     ├─ Filter besoins by scope, type, horizon
│     ├─ Run CommandeOFMatcher
│     └─ Build CandidateOF objects + BDH buffer OFs
│
├─ 5. Dynamic capacity calculation
│     ├─ Per-line: target_hours / active_days * 1.10
│     └─ Cap at 14h/day
│
├─ 6. Pre-assign target days
│     ├─ Historical profile → assign to peak weekday
│     └─ Otherwise → round-robin across workdays
│
├─ 7. Day-by-day scheduling loop
│     for each workday:
│       apply receptions for the day
│       apply incoming buffer additions
│       for each line:
│         GenericLineScheduler.schedule_day()
│
├─ 8. KPI computation
│     ├─ taux_service: % of orders served on time or from stock
│     ├─ taux_ouverture: planned_hours / effective capacity
│     ├─ deviation_penalty: ratio of jumped-ahead OFs
│     ├─ jit_penalty: penalty for exact-due-date scheduling
│     └─ score = w1*service + w2*ouverture - w3*deviation - w4*jit
│
└─ 9. Output via reporting.write_outputs()
      ├─ planning_{line}.csv per line
      ├─ stock_BDH_projete.csv
      ├─ ofs_non_faisables.csv
      ├─ lignes_commande_statut.csv
      ├─ kpis.json
      └─ alertes.txt
```

### 7.3 Day-Level Scheduling (`lines.py` — `GenericLineScheduler.schedule_day()`)

**Algorithm**:
1. Filter unscheduled candidates with positive charge
2. **JIT deferral** (non-last-day): Split into "active" (due ≤1 day) and "deferred". Pull deferred only if active hours < max(7h, 50% capacity)
3. **Greedy loop**:
   - Sort remaining by `generic_sort_key` (12-tuple)
   - Find first candidate that fits capacity (+2.5h tolerance)
   - Check material availability via `availability_status` (tries J-2, J-1, J)
   - Check BDH buffer availability
   - Assign time slot (with 15min setup if article changes)
   - Update buffer, kanban, family counts
   - Reserve components in material state
4. **Under-capacity gate**: If planned < min_open_hours → cancel all assignments

### 7.4 Priority Heuristics (`heuristics.py`)

12-tuple sort key (lower = higher priority):

| Pos | Factor | Logic |
|-----|--------|-------|
| 0 | `priority` | 0=BDH shortage, 1=normal, 2=BDH no shortage |
| 1 | `due_urgency` | 0=overdue, 1=tomorrow, 2=in 2 days, 3=not urgent |
| 2 | `jit_bonus` | -2 if due exactly today |
| 3 | `prematurity` | Days until due (penalizes early scheduling) |
| 4 | `target_day_delta` | Distance from assigned target day |
| 5 | `due_date` | Exact due date |
| 6 | `-charge_hours` | Larger OFs first |
| 7 | `serie_bonus` | -2=same article, -0.5=shared BOM, 1=different |
| 8 | `mix_penalty` | 1 if family overrepresented |
| 9 | `kanban_penalty` | Cumulative consumption / 50 |
| 10 | `article` | Tiebreaker 1 |
| 11 | `num_of` | Tiebreaker 2 |

### 7.5 Material State (`material.py`)

Virtual stock management for scheduling:
- **`build_material_stock_state(loader) → StockState`**: Initialize from real stock
- **`apply_receptions_for_day()`**: Add supplier deliveries for a given day
- **`reserve_candidate_components()`**: Virtually reserve scarce purchase components (stock < 2x requirement)
- **`availability_status(checker, loader, candidate, day) → ("comfortable"|"tight"|"blocked", reason)`**: Three-tier check at J-2, J-1, J
- **`tracked_bdh_requirements()`**: Recursive BOM walk computing BDH component needs

## 8. Reports (`production_planning/reports/`)

### 8.1 Action Report (`action_report.py` — 1449 lines)

The most comprehensive report. Produces **4 complementary views**:

1. **Component Actions** — One line per blocking component (missing qty, impacted OFs/commands, urgency classification)
2. **Supplier Actions** — One line per supplier/purchase-order (delayed receptions)
3. **Poste de Charge Risk** — Workstations at risk of shutdown
4. **Kanban Risk** — Feeder workstations below coverage threshold

**Urgency classification**: RETARD_FOURNISSEUR > AUCUNE_COUVERTURE > COUVERTURE_TARDIVE > SURVEILLANCE

**Kanban analysis**: Computes stock coverage in days, compares to threshold based on BOM depth and handoff delays.
